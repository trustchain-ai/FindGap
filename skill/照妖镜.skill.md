# 照妖镜 · Mirror Skill v0.5 · 11 招

> **一句话**：穷尽 171 条 Gap 根因（10 维度），取 ROI Top 11，一招 grep / 模板定位，照出原形句，PASS 才能进入下一流程。
>
> **形态承诺**：小而美。一份 SKILL.md + 几个真实示例。不写 CLI、不造 schema、不发明分级。Agent 自带 grep 能力，规则在表里直接读即可。

---

## 红线（5 条，永不违反）

1. ❌ **PASS/FAIL 二值**——不引入分数、严重度、加权评分
2. ❌ **杜绝虚假**——不编造数字 / 不立未用上的协议位 / 不写未发生的采纳故事
3. ❌ **不超 11 条规则**——v0.5 锁定 ROI Top 20%，新增需先砍旧
4. ❌ **不依赖记忆抑制**——同产物再次提交仍全量扫；feedback.jsonl 只在产物范围内静默
5. ❌ **不替 agent 干活**——给规则不给检测器，给原形句不给 30 字补齐文案

---

## 触发条件

满足任一即激活：

1. 显式：`/照妖镜`、`照一下`、`审一下`、`mirror`
2. 语义：用户说"能进下一步吗 / 上线前看看 / 有没有遗漏 / 鸡同鸭讲了 / 总返工"
3. 流程交接：用户标记"PRD 评审 / 方案评审 / PR 评审 / agent 任务下达 / 跨团队交接"

**反触发**：写代码、调 bug、闲聊。激活前自问：用户在做事还是审事？只有审事才激活。

## 11 招规则表（核心 · v0.5 本体）

每条规则：**一招定位法（≤20 字）+ 原形句（揭示认知盲点）+ 补齐方向**。Agent 1 秒内可 grep / 模板检查执行。

| # | 代号 | 名称 | 一招定位法 | 原形句（给上游看） | 补齐方向 |
|---|------|------|-----------|---------------------|---------|
| **1** | **S-PERF** | 性能/时效形容词无指标 | 语义检查：扫描"快/高效/尽快/海量/高可用"等词，**用 LLM 判断该处是在"模糊使用"还是"描述/示例/引用"** → 仅标记模糊使用 | "你说要快，但快到几毫秒没说" | P95 / QPS / TPS / 截止日 + 数字单位 |
| **2** | **S-QUANT** | 全称量词 / 模糊边界 | 语义检查：扫描"所有/任何/全部/通常/一般"等词，**LLM 判断是否缺少边界枚举**（排除集合/范围/阈值） | "你说'所有用户'，到底含不含未激活/已注销/内部账号" | 枚举边界 / 排除集合 |
| **3** | **S-NFR** | 质量形容词无判据 | 语义检查：扫描"健壮/稳定/可靠/安全/完善/友好"等词，**LLM 判断所在上下文是否缺少量化判据**（MTBF/错误率/SLA/威胁模型） | "你说'要稳定'，但崩溃率 0.5% 算稳定吗？" | MTBF / 错误率上限 / 威胁模型 / SUS 评分 |

> **灰度数据（dogfood/runs/004）**：本版灰度跑出**纯 grep 误报率 96.7%**——规则表自身包含了示例词（"你说要快"）、CHANGELOG 列出了主观词黑名单、references/ 旧文件含 G1 描述。已从纯 grep 升级为 **LLM 语义判断**——这正是 agent 自带的能力，无需工程化。详见 `../dogfood/runs/004-self-v05-grayfirst.md`。
| **4** | **R-DOD** | 验收标准（DoD）缺失 | 全文无 `验收\|DoD\|done when\|通过条件\|测试用例`，或验收项含主观词 | "你没定义什么叫做完" | 至少 1 条可测试 AC + 数值阈值 |
| **5** | **G-WHY** | WHY 缺失 / Output 冒充 Outcome | 全文无 `为了\|目标指标\|outcome\|北极星`，仅"做一个 X" | "你说做 X，但解决谁的什么问题没说" | 上位目标 + 受益方 + 量化预期收益 |
| **6** | **G-NOGO** | 多目标冲突 / 反目标缺失 | 全文无 `Non-goals\|不做\|out of scope\|优先序\|trade-off` | "你又要快又要便宜又要稳——选哪个？" | Non-goals 列表 / 优先序声明 / trade-off 取舍记录 |
| **7** | **I-SSOT** | SSOT 缺失 / 版本术语漂移 | 同一字段/术语在 ≥2 处定义不一致；或出现"以最新群里发的为准" | "你不知道这条信息谁是真相" | 单一权威源（文件路径或字段）+ 变更通知机制 |
| **8** | **I-ADR** | ADR 缺失 / 部落知识 | 关键 trade-off 段落无"为什么不选 X / 备选方案 / 决策时间" | "你忘了 trade-off 是怎么选的" | ADR 三段（选项 / 选择 / 理由）+ 决策人 + 日期 |
| **9** | **V-NAME** | 同名异义（鸡同鸭讲根源） | 关键名词（用户/事务/上线/完成/客户）无 glossary 锚定；同词跨段含义不同 | "你和他说的不是同一件事" | 术语表（含定义 + 反例）+ 跨文档一致性引用 |
| **10** | **V-STAKE** | 利益相关方 / 失败路径缺位（盲人摸象）| 评审参与方未含 SRE + 法务 + 客服任一；无回滚/工单/合规段落 | "你只看了 happy path，没人替失败场景说话" | Stakeholder 清单 + Failure path 段落（回滚 / 客服 / 合规）|
| **11** | **V-LAYER** | 抽象层 / 颗粒度错配（对牛弹琴根源）| 同段混用 OKR（战略词）+ 字段/参数级（实现词）；或同文档跨 3 个抽象层无标记 | "你在为什么层，他在怎么做层" | 在同一层对齐（Why / What / How 标注分段）|

**11 条覆盖的根因**（来自 171 条三维筛选，行业证据见 `dogfood/runs/003-self-v05-decision.md`）：

- **主观词**（S-PERF / S-QUANT / S-NFR）→ 用户原话"快、高"全覆盖；Berry / INCOSE 经典警示
- **需求清晰度**（R-DOD）→ Boehm 1981 / Capers Jones 验收模糊占缺陷 56%
- **目标层级**（G-WHY / G-NOGO）→ Feature Factory / Output ≠ Outcome 反模式
- **信息源**（I-SSOT / I-ADR）→ SSOT 缺失 + ADR 缺失，决策反复重开
- **视角**（V-NAME / V-STAKE / V-LAYER）→ 鸡同鸭讲 / 盲人摸象 / 对牛弹琴 用户原话全覆盖

## PASS/FAIL 判定（绝对二值）

```
11 条规则零命中  →  PASS  →  可流转下游
任一命中        →  FAIL  →  原形展示 + 退出本流程，待补齐后重照
```

**无中间态**、**无分级**、**无评分**。

---

## 原形展示模板（固定 5 字段）

每条 FAIL 必须按此格式输出，agent 直接复用：

```
[FAIL] <代号> <名称>
  file: <相对路径>:<行号>
  原形: "<逐字引用命中行>"
  why : <一句话说明为什么是 gap>
  补齐: <补齐方向，含数字/单位/边界提示>
```

**示例**（来自 examples/）：

```
[FAIL] S-PERF 性能/时效形容词无指标
  file: docs/prd/login.md:42
  原形: "登录要快"
  why : "快"是主观词，前后端对"快"理解差 10×（前端 100ms vs 后端 1s）
  补齐: 把"快"改为"登录接口 P95 ≤ 300ms（口径：从点击到首屏渲染完成）"
```

**禁止**：在 `[FAIL]` 块外加入额外评论 / 评分 / 严重度标签。固定字段即一切。

## 4 阶段流水线（保持 v0.4 简化，砍 Phase 0 引擎）

```
触发  →  Phase 1 扫描  →  Phase 2 输出  →  Phase 3 反馈
```

### Phase 1：扫描（≤1s）

对产物**顺序执行 11 条规则**：

1. 对每条规则的 grep/模板检查，命中即收集（位置 + 命中行）
2. **不评分**、**不分级**、**不加权**
3. 多文档评审时，跨文档检查仅适用于 I-SSOT / V-NAME

### Phase 2：输出（精简版 + 详细版，砍 v0.4 JSON 通道）

**精简版**（≤12 行，给人快速决策）：

```
照妖镜结论：N 条规则命中，本次 FAIL
- S-PERF × 2 / V-NAME × 1 / R-DOD × 1
下一步：按原形句补齐 4 处，重跑直到 PASS。
```

**详细版**（每条 FAIL 一个原形块，给上游逐项修订）：

```
[FAIL] <代号> <名称>
  file / 原形 / why / 补齐
```

**理由**：v0.4 的 JSON 第三通道已砍——agent 间结构化读取成本为 0，markdown 块本身就可被任意 agent 解析。

### Phase 3：反馈（保持 v0.3 反馈钩子，简化）

用户对任一 FAIL 可回复：

- `accept <代号>` → 接受，待修复
- `reject <代号> <理由>` → 误报，本产物范围内静默
- `fixed <代号>` → 已修复，请重照

写入产物根目录 `.zhaoyaojing/feedback.jsonl`（v0.3 已建机制保留）。**红线 4：feedback 只在产物范围内静默，不跨产物记忆**。

## 触发流程（端到端）

```
1. 上游用户写完 PRD / 方案 / 任务（A 流程结束时刻）
2. 调用：「照一下」or `/照妖镜 <产物>`
3. 照妖镜按 11 条规则扫，输出 PASS 或 N 个 [FAIL] 块
4. 上游按原形句 + 补齐方向修订
5. 重跑直到 PASS
6. PASS → 移交下游（B 流程接手）
```

---

## 杜绝什么（明确边界 · jq 哲学）

照妖镜**不做**以下事——保持小而美：

- ❌ 发散需求（那是 brainstorm skill 的事）
- ❌ 写实现方案（那是 planning skill 的事）
- ❌ 跑代码 / 起服务 / 调 LLM（agent runtime 自带）
- ❌ 维护跨产物记忆 / 学习用户偏好（违反红线 4）
- ❌ 评分 / 分级 / 排序（违反红线 1）
- ❌ 替上游写补齐文案（违反红线 5）
- ❌ 处理超出 11 条规则的根因（v0.6+ 才考虑扩展）

## v0.4 → v0.5 砍掉清单（不可逆决策）

| 砍掉 | 理由 |
|------|------|
| Phase 0 输入门禁引擎 | agent 自判输入充分性，引擎是过度工程 |
| INVEST-Agent 六要素 + readinessScore 0-100 | 二值就够，分数是虚假精度 |
| G1-G18 共 67 检测信号 | 11 条覆盖 80% 返工，其余 v0.6+ |
| severityScheme（P0/P1/P2/P3）| PASS/FAIL 二值即可 |
| gapIdFormat G{family}-{idx}#{hash} | 直接用 S-PERF/V-NAME 代号 |
| gate-output.schema.json + JSON 通道 | agent 结构化读取成本为 0，markdown 即够 |
| remediation-card 四选项卡片 | 列模板=替 agent 干活 |
| compatibility 三级表 | 单 skill 无兼容矩阵需求 |
| selfCheck C1-C6 | 11 条规则本身即 self-check |
| 双轨自检 5 项 | 红线 5 条已替代 |

完整决策推理见 [`../dogfood/runs/003-self-v05-decision.md`](../dogfood/runs/003-self-v05-decision.md)。

## 自我灰度策略（v0.5 起持续）

照妖镜把本项目自身作为**首个真实灰度用户**：

| 触发 | 自动产生 dogfood |
|------|----------------|
| 每次 PR / commit 前 | 扫所有改动 `*.md` → 写入 `dogfood/runs/` |
| 每次发布版本前 | 全量自照 → 写入 `dogfood/runs/` |

**目标指标**（v0.6 退出条件之一）：
- 误报率 ≤ 20%（用户标记 reject 的占比）
- 单次扫描定位耗时 ≤ 1s
- 11 条规则覆盖率 ≥ 80%（实际 FAIL 中 11 条之外的根因占比 ≤ 20%）

**红线**：dogfood 数据严格服务于本项目迭代，**不**回流在线决策。

---

## 给 Agent 的执行指令

1. 触发后**直接进入 Phase 1 扫描**，不解释 Skill
2. 11 条规则**逐条执行**，不跳条：
   - **S-PERF / S-QUANT / S-NFR**（#1-3）→ **LLM 语义判断**（不是 grep）：扫描主观词 → 判断是在"模糊使用"还是"描述/示例/引用" → 仅标记模糊使用
   - **其余 8 条**（#4-11）→ **grep / 模板检查**：grep 命中的行判断上下文语义 → 真正缺才报
3. **PASS/FAIL 二值**，禁止输出"基本通过""轻度命中"等中间态
4. 每条 FAIL **固定 5 字段**（file / 原形 / why / 补齐 + 代号名称），不加评论
5. **同产物多次提交仍全量扫**（红线 4）
6. **不发明新规则**——v0.5 锁定 11 条，新规则走 v0.6 issue 流程
7. **不 grep 规则表自身**：扫描时跳过"照妖镜"规则定义段（grep `## 11 招规则表` 到 `## PASS/FAIL` 之间的 CODE-FENCE 块）
