# 照妖镜 · Mirror Skill v0.1

> **核心使命**：在用户和 AI 之间识别 gap → 推荐最优消除手段 → 多轮收敛到对齐。

---

## 触发条件

满足以下**任一**即激活照妖镜流程：

1. 用户显式调用：`/照妖镜`、`照一下`、`mirror`
2. 用户提交产物并问"对不对/有没有问题/有没有遗漏/什么没想到"
3. 用户在评审场景（PRD / PR / 方案 / 设计文档 / 合同 / 决策）
4. 用户主动说"我觉得我可能漏了什么"

**不要在普通对话、闲聊、单纯写代码任务中激活**。激活前先问自己：用户现在是在"做事"还是在"审事"？只有"审事"才激活。

---

## 核心运行原理：4 阶段流水线

每次激活都严格按下面 4 阶段顺序执行。**不允许跳过、不允许并行**。

```
[1] Detect 识别 →  [2] Decide 决策 →  [3] Eliminate 消除 →  [4] Converge 收敛
```

### 阶段 1：Detect（识别 gap）

#### 第一步：分类产物类型

读用户提交的内容，归到下列其中之一：

| 产物类型 | 识别关键词 |
|---------|----------|
| `PRD/需求` | 需求、PRD、用户故事、功能描述、business requirement |
| `技术方案` | 架构、技术方案、设计文档、tech spec、design doc |
| `代码/PR` | diff、git、PR、代码、function、class |
| `决策/选型` | 选型、对比、方案 A vs B、决策、trade-off |
| `合同/协议` | 合同、协议、SLA、契约、terms |
| `对话/沟通` | 聊天记录、纪要、会议、邮件 |
| `其他` | 不属于以上任何一种 |

#### 第二步：扫描 gap 候选

按产物类型，从下面**8 种 gap 模式**中检测命中项：

| Gap 编号 | Gap 名称 | 检测信号 | 严重度 |
|---------|---------|---------|-------|
| **G1** | **主观锚词未量化** | 出现"快、好、友好、主流、合理、足够、大致、差不多"等无量化词 | 🟡 中 |
| **G2** | **隐含假设未声明** | 缺失"前提是 / 假设 / 给定"等假设声明，但讨论涉及边界条件 | 🔴 高 |
| **G3** | **角色心智漂移** | 同一概念在产物不同位置或不同角色（客户/产品/研发/测试）的描述不一致 | 🔴 高 |
| **G4** | **未问的关键问题** | 产物本应回答但未触及的常见关键维度（错误处理 / 性能 / 安全 / 回滚 / 合规） | 🔴 高 |
| **G5** | **来源缺失** | 出现"最佳实践 / 行业标准 / 通常 / 一般"等断言但无具体引用 | 🟡 中 |
| **G6** | **反例缺失** | 给出方案但未讨论失败场景 / 边界 case / 退化情况 | 🟡 中 |
| **G7** | **不可证伪** | 出现"提升用户体验 / 优化性能 / 增强稳定性"等无法验证的承诺 | 🟡 中 |
| **G8** | **责任真空** | 涉及多方协作但未指定 owner / 决策权 / 升级路径 | 🔴 高 |

**输出格式（标准化）**：

```markdown
## 🔍 Phase 1: Detect 识别结果

**产物类型**：{PRD/技术方案/...}
**扫描发现的 gap**：

| # | Gap | 类型 | 严重度 | 命中位置 | 命中证据 |
|---|-----|------|-------|---------|---------|
| 1 | G1 主观锚词未量化 | 人-人 | 🟡 | "性能要好" | "好"无量化 |
| 2 | G2 隐含假设未声明 | 人-AI | 🔴 | 全文 | 未说明并发量、数据规模 |
| ...
```

**关键约束**：
- 至少扫 8 种 gap 全部走一遍（不能漏）
- 每个命中必须给"命中位置"和"命中证据"（不能空口说有 gap）
- 如果某 gap 检测不到，明确写"未检测到"（不能省略）

---

### 阶段 2：Decide（决策最优消除手段）

#### Gap × 手段决策矩阵

| Gap 类型 | 推荐手段（按 ROI 排序） |
|---------|---------------------|
| **G1 主观锚词未量化** | 1️⃣ 直接给 3 档量化候选（如"实时 = 5s/1s/200ms 任选"）<br>2️⃣ 调出行业标准（如 SLA、SLO 基线）<br>3️⃣ 反向追问"如果不达到 X 会怎样" |
| **G2 隐含假设未声明** | 1️⃣ 列出常见前提清单让用户确认<br>2️⃣ 用 web search 找同类场景的典型假设<br>3️⃣ 用反例反推（"如果不是这样会怎样"） |
| **G3 角色心智漂移** | 1️⃣ 对比四方（客户/产品/研发/测试）对同一词的理解<br>2️⃣ 抽出最歧义的 3 个词强制澄清<br>3️⃣ 用统一定义表（glossary）补齐 |
| **G4 未问的关键问题** | 1️⃣ 从领域 checklist 自动补全（错误/性能/安全/回滚/合规 5 大维度）<br>2️⃣ 用 web search 找同类产物的"应有章节"<br>3️⃣ 调用 Claude 跨域知识合成 |
| **G5 来源缺失** | 1️⃣ 强制要求每条断言带 URL / 引用<br>2️⃣ web search 补全可信来源<br>3️⃣ 标注"来源缺失"，由用户判定是否需要 |
| **G6 反例缺失** | 1️⃣ 自动生成 3 类反例（边界 / 冲突 / 缺位）<br>2️⃣ Pre-mortem：假设方案失败了，倒推 5 个原因<br>3️⃣ web search 找同类方案的历史失败案例 |
| **G7 不可证伪** | 1️⃣ 把承诺转成量化指标（"提升 X" → "X 从 a 提到 b"）<br>2️⃣ 列出验证方法（怎么测、由谁测、什么算通过）<br>3️⃣ 如无法量化则标记"建议删除或重写" |
| **G8 责任真空** | 1️⃣ 强制 RACI 表（谁负责 / 谁批准 / 谁咨询 / 谁知会）<br>2️⃣ 明确升级路径（出问题找谁）<br>3️⃣ 标注决策点（谁签字算数） |

#### 选最优手段的决策规则

```
对每个识别出的 gap：
  IF gap 的严重度 == 🔴 高 AND 产物类型 ∈ {PRD, 技术方案, 合同, 决策}
    → 用矩阵推荐的第 1 个手段（最高 ROI）
  ELIF gap 的严重度 == 🟡 中
    → 在第 1 / 第 2 手段中选成本低的（优先规则匹配，不用 LLM）
  ELSE
    → 标注但不强制消除（用户决定要不要处理）
```

**输出格式**：

```markdown
## 🎯 Phase 2: Decide 决策

| Gap | 推荐手段 | 选择理由 |
|-----|---------|---------|
| G1 | 直接给 3 档量化候选 | 严重度中，规则可解，无需 LLM 介入 |
| G4 | 领域 checklist 自动补全 | 严重度高，PRD 必须有错误处理章节 |
| ...
```

---

### 阶段 3：Eliminate（执行消除）

对每个 gap 的推荐手段，调用对应能力执行：

| 手段类型 | 调用什么 |
|---------|---------|
| **规则匹配** | Skill 自带模板，直接产出（0 成本） |
| **web search** | 调 Claude 的 WebSearch 工具 |
| **跨域合成** | Claude 直接推理（不调外部） |
| **反例生成** | Claude 直接生成（按模板） |
| **Pre-mortem** | Claude 角色扮演"未来已失败的复盘者" |
| **领域 checklist** | 加载 Skill 内置 checklist（见附录） |

**执行约束**：
- 每个手段最多耗 30 秒
- 如果某手段执行失败，自动 fallback 到下一个手段
- 不允许"无限调用"——单次 Eliminate 最多 5 次外部调用

**输出格式**：

```markdown
## 🛠️ Phase 3: Eliminate 消除执行

### Gap G1：主观锚词未量化
**手段**：直接给 3 档量化候选
**执行**：
- "性能要好" 建议改为：
  - A) 99% 请求 < 200ms（高标准）
  - B) 99% 请求 < 1s（中等）
  - C) 95% 请求 < 5s（基本）
- 默认推荐：B（最常见 SLO）

### Gap G4：未问的关键问题
**手段**：领域 checklist
**执行**：你的 PRD 漏了下面这些章节：
- ❌ 错误处理（用户输入异常怎么办）
- ❌ 回滚策略（上线出问题如何回退）
- ❌ 性能边界（并发上限、超时）
...
```

---

### 阶段 4：Converge（多轮收敛）

#### 收敛规则

```
计算残留 gap 严重度总分：
  high_count × 3 + medium_count × 1 = severity_score

IF severity_score == 0
  → ✅ 完全对齐，输出最终报告
ELIF severity_score ≤ 2 AND 当前轮数 ≥ 2
  → ⚠️ 接近对齐，输出"可接受"报告 + 残留 gap 清单
ELIF 当前轮数 >= 3
  → 🛑 达到最大轮数，输出当前状态 + 标记"建议人工介入"
ELSE
  → 🔄 触发下一轮 Detect，重新走一遍
```

**关键约束**：
- 最多 3 轮（不允许无限循环）
- 每轮结束输出当前状态，让用户知道进度
- 每轮的 gap 必须减少（如果没减少，说明手段无效，主动告知用户）

**输出格式**：

```markdown
## 🎯 Phase 4: Converge 收敛

**第 1 轮结果**：
- 初始 gap 数：8（高 4 / 中 4）
- 消除后残留：3（高 1 / 中 2）
- severity_score：5

**判定**：继续第 2 轮

**第 2 轮结果**：
- 残留 gap 数：3 → 1（中 1）
- severity_score：1

**判定**：✅ 接近对齐，结束

---

## 📋 最终对齐报告

**对齐前**：8 个 gap，severity_score = 16
**对齐后**：1 个 gap，severity_score = 1
**对齐率**：93%

**残留 gap**（不阻塞但建议处理）：
- G5 来源缺失：方案中"业界普遍做法"无具体引用

**建议下一步**：
- 如果是决策性产物（合同/上线方案）→ 补全后再过照妖镜
- 如果是探索性产物（草稿/想法）→ 当前已可推进
```

## 附录：领域 Checklist（嵌入式知识库）

### A1. PRD/需求 必检 checklist

- [ ] 业务目标（解决什么真实问题）
- [ ] 用户画像（谁会用）
- [ ] 验收标准（怎么算做完）
- [ ] 边界条件（极端输入怎么办）
- [ ] 错误处理（出错怎么办）
- [ ] 性能要求（量化指标）
- [ ] 合规要求（隐私、安全、监管）
- [ ] 优先级 / 不做什么（明确 out of scope）

### A2. 技术方案 必检 checklist

- [ ] 架构图 + 数据流
- [ ] 关键技术决策 + 替代方案对比
- [ ] 性能 / 容量预估
- [ ] 失败模式 + 回滚方案
- [ ] 监控 / 告警 / 日志
- [ ] 安全设计（认证、授权、加密）
- [ ] 灰度发布策略
- [ ] 影响范围 + 兼容性

### A3. 代码 PR 必检 checklist

- [ ] 单测覆盖（关键路径）
- [ ] 边界 case 处理
- [ ] 错误返回 + 日志
- [ ] 性能影响（是否引入慢查询、N+1）
- [ ] 安全审查（输入校验、SQL 注入、XSS）
- [ ] 向后兼容（API/DB schema）
- [ ] 回滚方法

### A4. 决策/选型 必检 checklist

- [ ] 选项至少 3 个（不是单选题）
- [ ] 评估维度明确（成本/性能/可维护/...）
- [ ] 每个选项的反例（什么时候不该选这个）
- [ ] 推荐选项 + 推荐理由
- [ ] 决策可逆性（如果错了能不能改）

---

## 关键不变量（红线）

无论什么情况下，照妖镜都**不会**：

1. ❌ 给出没有"命中证据"的 gap（杜绝玄学）
2. ❌ 推荐"做一个 N 层架构"类的过度设计
3. ❌ 让用户在 ≥4 个选项里选（违反"带答案让用户确认"原则）
4. ❌ 跳过任何一个阶段（必须按 4 阶段顺序）
5. ❌ 把"未检测到"伪装成"已对齐"（必须诚实标注）
6. ❌ 超过 3 轮（防止无限循环）

---

## 给 Claude 的执行指令

当本 Skill 被触发时：

1. **不要先解释 Skill 是什么**，直接进入 Phase 1 开始扫描
2. **每个 Phase 结束必须输出该 Phase 的标准化报告**（用上面给的格式）
3. **不要询问用户"你想用哪个手段"**——按决策矩阵自动选最优
4. **遇到 Phase 3 执行失败时，主动 fallback 到下一个手段**，不要停下来问用户
5. **Phase 4 结束后，必须输出"对齐前 vs 对齐后"对比 + 残留 gap 清单**
