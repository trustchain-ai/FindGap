# FindGap · 1.0.0 · 交接点 Gap 校对器

> **一句话**：当产出物所有者认为内容已达到自己认知内的最优水平、准备交给下一个人或 agent 时，FindGap 识别并校对那些仍未被看见、但会在下游引发高成本返工的关键 gap。
>
> **三节点**：① 扫（受众检测 + 严重度判定）→ ② 查（WebSearch + 上下文推理兜底）→ ③ 出结果（Flat List 输出）。多一步就是多余。
>
> **定位**：FindGap 不是每轮对话前的 prompt checker，而是 handoff 前最后一道认知校对。它服务的是交接时刻：当 owner 已经觉得内容差不多、准备交给下一个人或 agent 继续推进时，优先识别那些会在下游放大成返工的关键 gap。

---

## 触发条件

用户说 `/findgap` `/fg` `/照` `/照妖镜` `照一下`。只在**准备交接给下一个人或 agent 前**扫描当前产出物、需求、方案或意图，做最后一次 handoff 校对。不扫代码、不扫 PR、不扫对话记录、不扫 AI 已交付的输出。

## 双模式

### Self-check

适合 owner 自己在 handoff 前做最后一次自检，重点看：
- 哪些默认前提只存在于自己脑中，没有显式写出
- 哪些地方自己以为讲清了，但 receiver 未必能按同样方式理解
- 哪些 gap 已经超出自己当前认知范围，因此不会再主动追问

### Receiver-check

适合接头人或下一个 agent 在接手前做他检，重点看：
- 如果现在开工，会卡在哪些信息缺口上
- 哪些地方会迫使自己默认补脑，从而带着错误认知继续做
- 哪些 gap 会在执行、联调、验收或跨角色协作中放大成返工

双模式只改变观察视角，不改变内部 11 条规则、三节点流程和输出契约。

## 内部扫描引擎（11 条规则，不对外展示代号）

以下规则在内部驱动 gap 检测，但输出中**只用人话描述 gap，不展示规则代号**。

| 内部代号 | 扫什么 |
|---------|--------|
| S-PERF | "快/高效/实时"等词是否缺量化指标 |
| S-QUANT | "所有/任何/全部"等词是否缺边界 |
| S-NFR | "稳定/可靠/安全"等词是否缺判据 |
| R-DOD | 全文是否缺可验收的完成条件 |
| G-WHY | 是否只说做什么没说为什么（stakes/DoD/priority 已足够解释时不触发）|
| G-NOGO | 多目标冲突是否缺优先序或"不做"列表 |
| I-SSOT | 同一对象是否在多处定义不一致 |
| I-ADR | 关键决策是否缺 why-not 记录 |
| V-NAME | 关键名词是否跨段含义不同 |
| V-STAKE | 是否缺失败路径/回滚/降级/受影响方 |
| V-LAYER | 同一决策点是否跨 Why/What/How 多层但未声明权威层 |

### 相邻规则边界

- G-WHY：问"为什么做"——stakes/DoD/priority 已足够解释时 suppress
- V-STAKE：问"错了谁受影响、会损失什么"
- G-NOGO：问"哪些不做、哪些优先级冲突还没选边"
- R-DOD：问"做到什么状态才算完成"
- V-LAYER：只在同一决策点跨层且未声明权威层时触发

## 三节点执行流程

### 节点 ① 扫

**受众检测**：

| 信号 | 受众 | 输出策略 |
|------|------|---------|
| 文档链接 + 审/靠谱/能不能开工 | 决策者 | 先致命项，其余折叠 |
| PRD/技术方案/合同/需求等关键字 | 评审者 | 先致命项 + 全量详情 |
| 一句话需求/日常意图描述 | 作者本人 | 全量平铺 |

**严重度判定**（每条 gap 必须标）：

| 严重度 | 定义 |
|--------|------|
| 🔴 致命 | 不改直接阻挡下游——接口/状态机/数据模型/架构会返工 |
| 🟠 阻塞 | 不改导致多人协作产生分歧 |
| 🟡 可延后 | gap 真实存在但不阻塞当前阶段 |

### 节点 ② 查

对每条 gap 做 **≥ 3 次 WebSearch**。

**三层证据体系**：

| 层级 | 来源 | 输出标记 |
|------|------|---------|
| **Tier 1** | WebSearch ≥2 条可点击来源互证 | 正常展示 URL |
| **Tier 2** | 搜不到公开数据，基于文档现状 + 领域常识推理 | `⚠️ 基于文档现状与领域常识推理：` |
| **Tier 3** | 连推理依据都不够 | `⚠️ 当前信息不足，需人工补充判断` |

**硬约束**：
- URL 必须真实可点击，404 即作废
- 不允许用训练数据伪装成搜到的
- 永远不停在"没数据"就结束——Tier 2 是兜底，不是放弃

### 节点 ③ 出结果（Flat List 输出）

**输出模板**：

```
FindGap · 发现 N 处 gap 可能导致返工

---

🔴 致命 · {用人话说的 gap 名}

原文："{被扫描文档里的原始句子}"

缺口：{直接说缺了什么、会怎么卡住}

优化方案：{基于行业数据/文档现状/领域常识给出的补齐方向，
说明为什么这样补比较合理}

参考依据：
- {来源}: {数据或结论} — {可点击 URL}
- {来源}: {数据或结论} — {可点击 URL}
（或）
- ⚠️ 基于文档现状与领域常识推理：{推理内容}

---

🟠 阻塞 · {用人话说的 gap 名}

原文："{原始句子}"

缺口：{直接说缺了什么}

优化方案：{补齐方向 + 决策依据}

参考依据：
- ...

---

🟡 可延后 · {用人话说的 gap 名}

原文："{原始句子}"

缺口：{直接说缺了什么，但不阻塞当前阶段}

参考依据：
- ...
```

**输出规则**：
- 🔴 和 🟠 必须有"优化方案"
- 🟡 可以省略"优化方案"
- 每条 gap 必须有"原文"引用
- 每条 gap 必须有"参考依据"（Tier 1 / Tier 2 / Tier 3）
- 不展示任何规则代号

---

## 红线（4 条）

1. **三节点上限**——不多于扫/查/出三步
2. **不落盘**——运行时不维护任何静态数据库，行业数据每次现查
3. **URL 真实**——编造 URL 一次 = 信用归零
4. **优化方案给方向不给结论**——提供决策依据和可选做法，但不替用户做最终选择

---

## 杜绝（明确不做什么）

- ❌ 给 PASS/FAIL
- ❌ 替用户改写文档
- ❌ 运行时维护静态知识库
- ❌ 扫 PRD/PR/对话记录/AI 已交付的输出（PRD/PR/对话属 v2.0+；已交付输出属 FindMiss）
- ❌ 输出中展示规则代号
- ❌ 多角色视角碰撞映射（已改为单一声音直述）
- ❌ 搜不到数据就停——必须用 Tier 2 兜底
