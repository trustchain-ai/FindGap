## v0.1 — 草稿（已完成）

**核心范围**
- 定义"反身性审视"骨架：触发词识别、单轮 gap 列表输出
- 最小工件：一份 prompt + 一个触发器表
- 仅在本地 IDE / Claude Code 内运行

**工期**：2 周（已结束）

**退出条件**
- 能识别"照一下/审一下/有没有遗漏"等核心触发短语
- 能输出 ≥3 条 gap 的非空列表
- 用户能在 1 轮交互内拿到首版反馈

**验收**
- 在 10 个内部 PRD / PR 评审场景中触发成功率 ≥ 80%
- 至少 1 名核心用户每周主动调用 ≥ 3 次

## v0.2 — Gap 识别能力大扩展（已发布）

**核心范围**
- gap 分类体系成型：事实 / 逻辑 / 范围 / 风险 / 盲区 5 类
- 多轮收敛：从"列 gap"升级为"列 gap → 排优先级 → 推荐消除手段 → 对齐"
- 触发场景扩展：PRD 评审、PR 评审、技术方案、对话决策、复盘
- 输出结构化：每条 gap 带 类别 / 证据 / 建议动作 / 置信度

**工期**：4 周（已结束）

**退出条件**
- 5 类 gap 分类在 prompt 中固化并稳定输出
- 多轮收敛 loop 默认开启，平均 ≤ 3 轮达成"对齐"
- 至少覆盖 5 类协作场景的触发模板

**验收**
- 内部 30+ 真实案例中，用户接受的 gap 占比 ≥ 60%
- 误报（用户标记"无效 gap"）比例 ≤ 25%
- 多轮收敛成功率（最终用户确认"已对齐"）≥ 70%

## v0.3 — 双轨报告模板 + dogfood 闭环（当前）

**核心范围**
- **双轨报告模板**：
  - 轨道 A — 协作者视角：给被审视的人，强调"你下一步该做什么"
  - 轨道 B — 反身视角：给审视的发起人，强调"你自己的预设有哪些被照出来"
  - 两轨在同一次调用中并列产出，互为镜像
- **dogfood 闭环**：
  - 每次调用结果回收：用户标注 gap 是否有效 / 是否触达盲区
  - 标注数据回流到 prompt 迭代，形成"用 → 反馈 → 改 prompt → 再用"周期
  - 维护一个 dogfood 记录表（场景 / 调用 / 接受率 / 改进点）

**工期**：6 周（进行中，当前为 Week 3）

**退出条件**
- 双轨模板在所有触发场景中默认输出
- dogfood 反馈回收机制可用（每次会话末尾的轻量打分 + 可选标注）
- 至少完成 1 轮基于回收数据的 prompt 迭代

**验收**
- 双轨同时被用户认为"有用"的比例 ≥ 50%
- dogfood 数据样本量 ≥ 80 次调用
- 基于回收数据的迭代，使 gap 接受率较 v0.2 提升 ≥ 10 个百分点

## v0.4 — Gate · Agent 门禁对齐（当前开发）

**核心定位转身**
FROM：「事后评审工具」——照出 gap 给人看
TO：「Agent 执行前的输入质量门禁」——拦截 60% 可预防失败（Salesforce 实测：prompt 质量优化将 agent 解决率从 40% → 84%）

**核心范围**

- **Phase 0 预检层**：在 Phase 1 Detect 之前，对任务输入做 INVEST-agent 六要素 + 槽位完整度评分；不通过则生成补齐卡片而非带伤扫描
- **G15-G18 四维新增**：
  - G15 任务可分解性（INVEST-S, Small）
  - G16 上下文完整度（业务术语 / 团队惯例 / 既有系统）
  - G17 示例充分性（输入/输出示例 ≥1 正 + 1 反）
  - G18 副作用可逆性（不可逆操作硬阻断）
- **结构化补齐卡片**：缺什么 + 为什么重要 + 4 选项怎么补 + 补完后自动重评
- **机读 JSON 输出通道**：在精简版/详细版之外新增第三通道，供 CI/agent 直接消费
- **readinessScore 评分体系**：0-100，阈值 80 放行；G14 五维 + G15-G18 四维加权
- **数据来源**：本项目自身迭代作为灰度，自我审视产出 dogfood 数据，自我驱动迭代

**工期**：6.5 周（2026-06-02 → 2026-07-14）

| 周次 | 交付物 | 验收 |
|------|--------|------|
| W1 | Phase 0 骨架 + readinessScore 公式 + INVEST-agent 规则 | mock 任务输出 verdict+score |
| W2 | G15-G18 入 gap-taxonomy.md + 补齐卡片模板 | 每维 ≥2 检测信号 + 改写模板 |
| W3 | JSON schema + 三通道输出集成 | 单次扫描产出三份输出 |
| W4 | Phase 0 ↔ Phase 1-4 联调 + 自动重评循环 | 端到端 BLOCKED → 补齐 → PASSED |
| W5 | Dogfood 前 10 案例（自身灰度为主） | 收集阻断准确率 + 补齐完成率 |
| W6 | Dogfood 后 10 案例 + 对照组比对 | 阻断准确率 ≥80% |
| W6.5 | 退出评审 + v0.4.0 发布 | 所有 SMART 退出条件通过 |

**退出条件（SMART）**

- **S**：交付 Phase 0 代码 + G15-G18 规则 + 补齐卡片模板 + JSON schema + readinessScore 公式
- **M**：20 个真实 agent 任务部署门禁：①阻断准确率 ≥80% ②首次成功率较对照组提升 ≥25 百分点 ③误阻断率 ≤15%
- **A**：基于 v0.3 已验证扫描框架，纯增量；单人 6.5 周可完成
- **R**：从「事后审」变「事前门禁」，直接推进终极目标
- **T**：2026-07-14 完成

## v0.5 — Gate Plus（计划，路线翻盘）

**核心范围**（取代原计划的"数据驱动人格 + 单平台轻接入"）

- **Agent-to-Agent 自主触发协议**：将 v0.4 的机读 JSON 包装为 pre-execution hook，使 Agent A 完成产出后可自动触发照妖镜审视，输出给 Agent B 作为上下文约束——无需人介入
- **CI/CD 集成**：GitHub Action + PR comment API；PR 触发 → 门禁扫描 → 内联评论 → 阻断/警告
- **单平台轻接入**：Hermes 或 Claude Code skill marketplace，使外部用户可一键安装+触发

**翻盘理由**：v0.3 自照暴露 0 条真实 dogfood，"数据驱动人格"无数据支撑；v0.4 的 JSON 输出天然就是 hook，v0.5 顺势把"被人调用"扩展为"被 agent 调用"，更贴近终极目标。

**工期**：8 周

**退出条件**

- A2A 协议在 ≥2 个 agent 框架（Claude Code skill / Hermes profile）中跑通
- CI 集成在本 repo 跑通，PR 评论可见门禁结果
- 单平台月活调用 ≥200 次
- 外部独立调用方 ≥3

## 终极目标（SMART 化版本）

**到 2027-12-31，照妖镜在 ≥3 个独立团队的 PRD / PR / 技术方案评审场景中，作为"agent 执行前置门禁"被默认启用。**

**可度量逼近信号**：

| 指标 | 基线（v0.3 末） | v0.4 目标 | v0.5 目标 | v1.0 目标 |
|------|----------------|-----------|-----------|-----------|
| 入 agent 前 readinessScore ≥80 的比例 | ~30%（估算） | ≥50% | ≥60% | ≥70% |
| 累计真实 dogfood 反馈条数 | 0 | ≥80 | ≥200 | ≥500 |
| 外部独立调用方数量 | 0 | ≥1（含作者自身灰度） | ≥3 | ≥10 |
| 自发调用占总调用比例 | 0% | 10% | 30% | 50% |
| Agent 执行首次成功率提升（vs 无门禁对照组） | — | ≥25 百分点 | ≥30 百分点 | ≥35 百分点 |

**为什么这是正确的目标**：
- "默认启用"是可观测行为（看是否进入流程）
- "团队"而非"用户"——团队验证比个人验证更接近真实采纳
- 五个量化指标构成网络效应递进：单点采纳 → 数据沉淀 → 跨团队复制 → 生态化
- 阶段性目标连贯，每个版本都能验证是否在正确轨道上

**砍掉的旧目标**："让'照一下'成为默认调用的反身性环节"——已确认 SMART 4/5 维 fail（v0.3 自照报告），此处替换。
