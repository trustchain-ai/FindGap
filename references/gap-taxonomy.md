# Gap Taxonomy — 详细扫描信号

> SKILL.md Phase 1 引用的完整 gap 扫描信号表。从 SKILL.md 拆分出来以控制主文件 ≤ 25KB。

> 扫描原则：宁错杀不放过；命中信号 → 立即报 gap，不要求 100% 把握。

## G14 SMART 双向扫描

### G14.S — Specific 具体性

**信号**：形容词式目标 / 范围模糊词 / 动词缺宾语 / 主体不明 / 目标嵌套
**严重度**：critical = 只有形容词无名词宾语 / high = 抽象集合需追问 / medium = 上下文可推断
**改写模板**：
```
在 [具体模块/页面/接口] 上，由 [责任方] 完成 [可指认动作]，
针对 [明确对象] 的 [可定义属性]。
```
**示例**：「我们要优化用户体验」→「前端团队在【订单详情页】的【退款入口】将【点击到结果展示】的【交互步骤】从 5 步减少到 2 步」

### G14.M — Measurable 可量化

**信号**：通篇无数字 / 伪量化词（显著/大幅）/ 缺基线 / 缺口径 / 体感词目标
**严重度**：critical = 无数字且涉及决策投入 / high = 有数字但缺基线 / medium = 缺过程指标
**改写模板**：
```
将 [指标名]（口径：[分子/分母/统计窗口]）从 [基线值] 提升/降低到 [目标值]，
过程指标观察 [二级指标]。
```
**示例**：「显著降低崩溃率」→「Android 客户端【启动崩溃率】（crash 次数 / 启动次数，7 日滑动窗口）从【0.8%】降至【0.2%】以下」

### G14.A — Achievable 可达成

**信号**：目标值断崖 / 无依赖盘点 / 资源失配 / 口号式 / 缺降级方案
**严重度**：critical = 物理不可能或绝对值 100%/0% / high = 缺关键依赖 / medium = 偏激进无 Plan B
**改写模板**：
```
基于 [历史基线/同类项目数据]，在 [资源 X 人 × Y 周 + 预算 Z] 下，
目标 [合理区间]；关键依赖：[依赖 1, 2]；
若 [风险项] 触发，降级到 [Plan B]。
```
**示例**：「下个月把崩溃率降到 0」→「下个月将 Top3 高频崩溃栈修复，崩溃率从 0.8% 降至 0.3%（历史最优 0.25%）；依赖 QA 提供 crash 聚类 + Bugly 接入；若 SDK 侧占比 >50%，降级到 0.5%」

### G14.R — Relevant 相关性

**信号**：无上位目标 / 技术指标未连业务结果 / 为做而做 / 跨目标冲突未化解
**严重度**：critical = 与上位完全无关 / high = 相关性弱或冲突未化解 / medium = 优先级模糊
**改写模板**：
```
本目标支撑 [上位目标/季度 OKR X]，预期对 [核心业务指标 Y] 贡献 [量化收益]；
主要受益方：[角色]；与 [其他目标 Z] 的优先级关系：[高于/低于/并列]。
```

### G14.T — Time-bound 有时限

**信号**："尽快/近期/下一步" / 只有终点无里程碑 / 节点无验收物 / 评审节点缺失
**严重度**：critical = 完全无时间约束 / high = 终点无里程碑且跨度 >2 周 / medium = 验收物模糊
**改写模板**：
```
截止 [YYYY-MM-DD]（自然/工作日），里程碑：
- M1 [日期+交付物+验收标准]
- M2 [日期+交付物+验收标准]
评审节点：[日期+形式]；
外部依赖到位时间：[依赖方+日期]；
缓冲：[N 天]。
```

## G15-G18 Agent 门禁专属维度（v0.4 新增）

> **背景**：v0.4 将照妖镜定位转身为「Agent 执行前的门禁对齐系统」，G15-G18 为现有 G1-G14 未覆盖的 agent 输入质量维度。
>
> **来源依据**：Salesforce 实测 prompt 质量优化将 agent 解决率从 40% → 84%；arXiv 研究 27-78% 的 agent "成功"包含损坏的推理链；执行前可拦截约 60% 失败。

### G15 — Task Decomposability 任务可分解性

**信号**：
- 任务串联多目标的连接词（"并且 / 同时 / 顺便 / 然后 / 此外"）出现 ≥2 次且各分句独立
- 单条任务覆盖多个产物类型（PRD + 代码 + 部署 + 验收）
- 子目标数 ≥4 但未拆分为子任务序列
- 复杂度估算超出单轮 agent 能力（如要求"重构 + 测试 + 部署 + 文档"一次完成）

**严重度**：
- critical = 单任务串联 ≥3 个独立目标且涉及不可逆操作
- high = 串联 ≥2 个独立目标
- medium = 复杂度高但目标单一

**改写模板**：
```
原任务："实现 X 并部署到生产并通知所有用户"
拆分为：
  T1. 实现 X（验收：单元测试通过 + PR review 通过）
  T2. 灰度部署 X（验收：1% → 10% → 100% 阶梯 + 监控指标正常）
  T3. 通知用户（验收：发送渠道 + 内容 + 时机明确）
依赖：T2 依赖 T1，T3 依赖 T2
```

**学术对照**：Atlassian INVEST-S (Small) + Addy Osmani "small focused context > giant prompt"

---

### G16 — Context Completeness 上下文完整度

**信号**：
- 任务出现业务术语但无定义（如"核销 / 回收 / 风控"等领域词无术语表）
- 涉及团队惯例但无规范文档引用（"按我们的标准方式" / "类似其他项目"）
- 引用既有系统但无架构说明或接口文档链接
- 假设用户已知前提（"如你所知"、"显然"、"通常"）但前提未列出
- 涉及外部数据但未提供数据样本或 schema

**严重度**：
- critical = ≥1 个核心术语未定义且影响主路径
- high = 关键前提隐含但未声明
- medium = 边缘上下文缺失但可推断

**改写模板**：
```
请补齐以下上下文槽位（每项二选一）：
  □ 核心术语：____ = ____（一句话定义）或 参见 [文档路径]
  □ 团队惯例：____ 按 [规范文档] / [既有代码风格] 处理
  □ 既有系统：架构图见 [链接]，关键接口 [接口名] 文档见 [链接]
  □ 数据样本：附 ≥1 条真实样本或测试 fixture
```

**学术对照**：Representational Incompleteness（NCBI 2022）+ 知识诅咒（G13.8 的执行落地版）

---

### G17 — Example Sufficiency 示例充分性

**信号**：
- 任务要求生成特定格式/风格产物但零示例（"按团队风格写"无示例）
- 复杂输出结构（JSON/YAML/SQL/特定 DSL）但无样例
- 边界条件涉及但无反例样本
- "类似 X" / "参考 Y" 但 X/Y 无具体引用

**严重度**：
- critical = 强格式约束但零示例（如要求生成 LangGraph 节点）
- high = 有 1 个正例但无反例
- medium = 示例 ≥2 但未覆盖边界

**改写模板**：
```
请附加示例（最少 1 正 + 1 反）：
  正例：____（一个符合预期的输入/输出样本）
  反例：____（一个明显错误的输入/输出样本，并说明错在哪）
  边界例（可选）：____（极端输入下的预期行为）
```

**学术对照**：tree-of-thought 比 naive prompting 成功率高 4-18 倍（arXiv 多篇）；Addy Osmani "2-3 个好示例胜过一页说明文档"

### G18 — Side-Effect Reversibility 副作用可逆性

**信号**：
- 任务涉及不可逆操作动词：删除 / 部署 / 支付 / 发送 / 上线 / 迁移 / 推送 / `DROP` / `rm -rf` / `git push -f`
- 涉及外部资源变更（数据库写入 / API 提交 / 文件覆盖）
- 涉及通知/营销/触达用户的发送类操作
- 操作影响范围跨服务、跨账户、跨地域

**严重度**：
- critical = 不可逆且无回滚预案（必须人工确认才放行）
- high = 不可逆但有回滚预案（强警告）
- medium = 可逆但代价高（如全量同步 1TB 数据）

**改写模板**：
```
本任务包含不可逆操作：____
请补齐以下任一：
  A) 回滚预案：如出错，[具体回滚步骤] + [回滚 SLA]
  B) 灰度策略：1% → 10% → 100% 阶梯 + 每阶段观察 ≥N 分钟
  C) 人工确认门：[姓名/角色] 在 [时间] 前确认后才执行
  D) 二次确认：执行前请求用户输入操作摘要确认
```

**学术对照**：fail-closed by default（业界共识）+ Data443 Pre-Execution Control Layer + AWS Bedrock AgentCore policy
