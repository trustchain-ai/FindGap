# 贡献指南

照妖镜是一份**操作手册（Skill）**，不是一个代码工程。贡献门槛极低——能改 Markdown 就能贡献。

---

## 你可以贡献什么

| 类型 | 难度 | 示例 |
|------|-----|------|
| **新增 gap 类型** | ⭐ | 当前 8 类，发现新的 gap 模式可加到 SKILL.md |
| **新增手段** | ⭐⭐ | 决策矩阵里某个 gap 推荐手段太弱，提供更好的 |
| **新增 examples** | ⭐ | 在 `examples/` 加你的真实案例 |
| **新增领域 checklist** | ⭐⭐ | 当前 4 类（PRD / 技术方案 / PR / 决策），可扩展到合同/法律/医疗等 |
| **多语言支持** | ⭐⭐ | 把 SKILL.md 翻译成英文 / 日文 |
| **修复触发误判** | ⭐⭐⭐ | 发现 Skill 在不该激活的场景激活，提供反例 |

---

## 提交流程

1. Fork 本仓库
2. 新建分支：`git checkout -b feat/add-gap-type-G9`
3. 修改 `skill/照妖镜.skill.md`（或对应 example/test）
4. **必须在 `examples/` 加 1 个真实 case 证明改动有效**
5. 提 PR

> **维护者注**：如果 `git push` 在沙箱 / 公司代理 / 受限环境下被 SIGKILL 或挂起，
> 用 [`scripts/api-push.sh`](scripts/README.md) 走 GitHub Git Data API 推送，
> 不再多次重试 `git push`。该脚本是 v0.4 发布过程的沉淀产物。

---

## PR 标准

| 改什么 | 必须证明 |
|-------|---------|
| 新增 gap | 给 ≥ 2 个真实命中证据 |
| 新增手段 | 解释为什么比现有手段 ROI 更高 |
| 修触发 | 给 ≥ 3 个 false positive / false negative 反例 |

---

## 沉淀真实 case

如果你在工作中用了照妖镜，**贡献你的 case 是最有价值的**：

1. 把"产物原文"放到 `examples/NN-描述.md`
2. 跑一遍照妖镜，把 4 阶段输出贴上
3. 标注哪些 gap 真有用、哪些是误报

这样积累的 case 库会反哺 SKILL.md 决策矩阵优化。

---

## 不接受的 PR

- ❌ 给 Skill 加更多功能但没有真实使用证据
- ❌ 把 Skill 变成代码（加 hook / 加 MCP / 加 Python）—— 这些属于 v0.3+ 的事
- ❌ 把 Skill 改成"问用户更多问题"的形态（违反"带答案让用户确认"原则）

---

## 行为准则

- 诚实评估：发现 Skill 失效的场景必须报告，不要包装成"功能不全"
- 反对过度设计：能用 1 段 Markdown 解决就不要加 1000 行代码
