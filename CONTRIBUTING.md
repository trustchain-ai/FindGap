# 贡献指南

照妖镜是一份**操作手册（Skill）**，不是一个代码工程。贡献门槛极低——能改 Markdown 就能贡献。

---

## 你可以贡献什么

| 类型 | 难度 | 示例 |
|------|-----|------|
| **改进 11 招规则** | ⭐ | 某条规则误报率高，提供更精准的定位法 |
| **新增 examples** | ⭐ | 在 `examples/` 加你的真实照镜子案例 |
| **多语言支持** | ⭐⭐ | 把 SKILL.md 翻译成英文 / 日文 |
| **修复触发误判** | ⭐⭐⭐ | 发现 Skill 在不该激活的场景激活，提供反例 |

---

## 提交流程

1. Fork 本仓库
2. 新建分支：`git checkout -b feat/improve-s-perf-rule`
3. 修改 `skill/照妖镜.skill.md`（或对应 example）
4. **必须在 `examples/` 加 1 个真实 case 证明改动有效**
5. 提 PR

---

## PR 标准

| 改什么 | 必须证明 |
|-------|---------|
| 改进规则 | 给 ≥ 2 个真实命中证据 |
| 修触发 | 给 ≥ 3 个 false positive / false negative 反例 |

---

## 沉淀真实 case

如果你在工作中用了照妖镜，**贡献你的 case 是最有价值的**：

1. 把你的 prompt 原文放到 `examples/NN-描述.md`
2. 跑一遍 `/照`，把三节点（照→查→亮）输出贴上
3. 标注哪些命中有用、哪些是误报

---

## 不接受的 PR

- ❌ 给 Skill 加评分/分级/PASS-FAIL（照妖镜只反射，不裁判）
- ❌ 把 Skill 变成代码（加 hook / 加 MCP / 加 Python）
- ❌ 加静态数据库/锚点库/知识库（行业基准每次现查）

---

## 行为准则

- 诚实评估：发现 Skill 失效的场景必须报告，不要包装成"功能不全"
- 反对过度设计：能用 1 段 Markdown 解决就不要加 1000 行代码
