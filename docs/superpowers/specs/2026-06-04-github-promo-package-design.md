# Zhaoyaojing GitHub Promotional Package Design Spec

## 1. Overview

This spec defines a GitHub-native promotional package for Zhaoyaojing designed to convert readers into first-time `/照` users, and secondarily into GitHub stars or follows.

The package should lead with visible user value, then explain method. The recommended structure is a two-layer asset system:

1. **README short version** for immediate GitHub conversion
2. **One long-form artifact** for deeper proof and persuasion
   - preferred formats: **GitHub Discussion** or **repo article**

The promotional package should draw from **6 sanitized, repo-derived cases** based on dogfood and example materials already present in the repository. It must avoid unsupported quantitative claims and preserve the project's core promise: reflection before send, not prompt rewriting.

## 2. Goals

### Primary Goal
Get readers to **try `/照` immediately** inside an existing AI workflow.

### Secondary Goals
- Earn GitHub stars from readers who identify with the problem
- Earn follows from builders who want to watch the method evolve
- Establish Zhaoyaojing as a credible, disciplined prompt-clarity artifact rather than a generic prompt-tips project

### Non-Goals
- Do not optimize for enterprise sales messaging
- Do not position the project as a benchmarked evaluation product yet
- Do not claim scientific accuracy, production precision, or broad external validation before a documented baseline exists
- Do not front-load manifesto or philosophy before user value is clear

## 3. Target Audience

### Audience A: AI Power Users
People who frequently write prompts for coding, PRDs, planning, or decision support.

**What they care about:** faster alignment, fewer bad AI assumptions, less prompt iteration.

**What converts them:** seeing that `/照` catches ambiguity they plausibly write themselves.

### Audience B: Agent / Platform Builders
People building agent workflows, prompt gates, review layers, or internal AI tooling.

**What they care about:** reusable method, skill form factor, inspectability, low integration friction.

**What converts them:** clear method boundaries, concrete examples, and a believable operating model.

### Audience C: Managers / Team Leads
People responsible for output quality, coordination cost, or AI adoption quality.

**What they care about:** preventing downstream misunderstanding, reducing vague requests, improving consistency.

**What converts them:** before/after reflection framing and concrete communication failures avoided.

### Audience Priority Order
1. AI power users
2. Agent / platform builders
3. Managers / team leads

The package should speak first to the person who can try `/照` alone in under 5 minutes.

## 4. Core Narrative

### Chosen Narrative
**Result-driven opening.**

Lead with the outcome readers want: catching ambiguity before hitting Enter. After the reader sees real cases, explain the 11-rule mirror and the public-source-backed reflection method.

### Narrative Formula
1. **You already write prompts like this**
2. **Those prompts hide ambiguity**
3. **`/照` reflects the ambiguity before send**
4. **The reflection is grounded in real cases and, when relevant, public-source lookup rather than made-up advice**
5. **Try it now, then learn the method if you care**

### Positioning Line
Zhaoyaojing is not a prompt rewriter, scorer, or judge. It is a **pre-send ambiguity mirror**.

### Required Framing
All promotional assets must frame improvement as:
- **ambiguity-before**
- **reflection-after**

They must not frame improvement as:
- bad prompt before / rewritten prompt after
- old prompt / fixed prompt generated by the tool
- AI rewrites your prompt into the correct version

This distinction is load-bearing because the project promise is reflection, not substitution.

## 6. Proof Strategy

### Proof Model
Use a **cases + method** dual track.

#### Track A: Cases
Show 6 sanitized ambiguity patterns derived from dogfood and example materials. The job of the cases is to make readers say: "Yes, I write prompts like this."

#### Track B: Method
Explain, briefly, how Zhaoyaojing works:
- scan with 11 ambiguity rules
- look up public benchmarks or references when relevant
- return reflection, not judgment

### Why Dual Track
- Cases alone feel anecdotal
- Method alone feels abstract
- Together, they create usability plus credibility

### Approved Proof Types
- Sanitized excerpts derived from real dogfood/example materials
- Directly stated rule names already used by the product
- Honest process descriptions grounded in repo materials
- Repo-native evidence such as examples, README positioning, and roadmap state

### Disallowed Proof Types
- Precision percentages not yet established in baseline data
- "X% more accurate" claims
- "Proven to reduce rework" claims unless backed by a published study or completed internal baseline
- Synthetic testimonials presented as real users
- Any implication that the tool always finds the right issue

### Trust Copy Guidance
Preferred phrasing:
- "Based on real dogfood runs"
- "Built from real prompt failures seen in practice"
- "Uses public-source lookup when relevant"
- "Does not claim certainty where none exists"

Avoid:
- "Scientifically validated"
- "High precision"
- "Guaranteed better prompts"
- "Accurate across models"

## 7. Sanitization Rules

All promotional artifacts must pass a sanitization review before publishing.

### Must Sanitize
1. **Merchant names**
2. **Internal system names**
3. **Internal product or code names**
4. **Internal links, document paths, ticket IDs, screenshots, URLs**
5. **Email addresses, names, owners, reviewer handles**
6. **Timestamps, commit metadata, trace IDs, scan IDs, hidden file metadata if exposed in screenshots or copied blocks**
7. **Org-specific workflow labels or internal jargon that could identify the source organization**

### Sanitization Method
- Replace company or merchant references with neutral descriptors such as "a payment flow," "a checkout system," "an internal workflow," or "a product spec"
- Replace internal tools with functional names, not aliases
- Strip all direct links unless they point to public repo assets or public sources
- Rewrite prompt excerpts only enough to anonymize identity while preserving the ambiguity pattern
- Remove metadata headers from case presentation unless they are already public and materially improve trust

### Preserve While Sanitizing
- The ambiguity pattern
- The type of work being described
- The conversational failure mode
- The reflection insight

### Sanitization Red Lines
Do not publish any case if anonymization would still allow a reasonable insider to infer the original company, merchant, team, or system. If the ambiguity pattern cannot survive sanitization without likely leakage, drop the case.

### Recommended Path
Ship in this order:

1. **README short version** as the conversion page
2. **One long-form artifact** as the proof page

### Why This Path
The README is where discovery traffic converts. The long-form artifact is where skeptics, builders, and share-driven readers go to understand the evidence and operating model.

### Channel Recommendation for the Long-Form Artifact

If only one long-form artifact is produced, choose the channel deliberately:
- prefer **GitHub Discussion** when feedback collection matters most
- prefer **repo article** when message polish and reusability matter most

## 10. README Short Version Structure

The README short version should optimize for quick comprehension and immediate trial.

### Recommended Structure
1. **Hero line**
   - one-line promise
   - one-line Chinese support line if bilingual positioning is retained

2. **Result-first example**
   - a short ambiguity-before snippet
   - a short reflection-after snippet
   - no metrics or performance claims

3. **What `/照` does**
   - 2 to 3 bullets max

4. **Why it's different**
   - no rewrites
   - no fake numbers
   - no scores
   - public-source lookup when relevant

5. **Try now**
   - one-sentence install or usage instruction
   - 3 usage examples max

6. **Trust signal by structure, not hype**
   - "Built from real dogfood cases"
   - "6 example ambiguity patterns"

7. **Secondary CTA**
   - star / follow / share case

### README Writing Constraints
- Make value obvious above the fold in under 15 seconds
- Do not overload methodology before the CTA
- Do not bury install or try instructions below long explanations
- Use one flagship case, not all 6
- Keep the first screen tight: target 120-180 Chinese characters of core value before the first break
- Favor short bullets over paragraphs; every line should carry proof, differentiation, or action

## 11. Long-Form Artifact Structure

The long-form artifact should deepen trust through layered proof.

### Recommended Structure
1. **Opening: result-driven hook**
   Example direction: "Most prompt failures start before the model answers."

2. **The user problem**
   Explain that many prompts fail because terms like fast, done, support, clear, or default mean different things to different readers.

3. **Flagship case**
   Start with the most relatable sanitized case, ideally the PRD review or vague performance target.

4. **The six cases section**
   Organize by ambiguity families, not chronology.

5. **What `/照` actually does**
   Brief method section: 11 rules, public-source lookup when relevant, reflection output.

6. **What it does not do**
   No scores, no rewrite theater, no unsupported precision claims.

7. **Why this shape matters for agent workflows**
   Explain why a reflection step before send matters for builders and managers.

8. **Why the project is intentionally narrow**
   Position the focus on prompt ambiguity, not a general AI evaluation platform.

9. **Call to action**
   Try `/照` on your next PRD, spec, or task request.

10. **Footer trust note**
   State explicitly that baseline work is still in progress and external precision claims are intentionally withheld.

### Suggested Section Pattern for Each Case
- ambiguity-before
- what `/照` reflected
- why that ambiguity matters
- optional public-source note, only when it materially strengthens the case

### Long-Form Writing Constraints
- Control overall length: default target 1,000-1,600 Chinese characters or equivalent dense markdown
- Prefer 3 flagship cases in full and compress the remaining cases into a pattern summary
- Every paragraph should either add proof, explain method, or drive trial; cut scene-setting that does none of the three
- Avoid long industry exposition unless it directly sharpens the case

## 14. What Not to Say

These are hard prohibitions for all promotional artifacts.

### Do Not Claim
- precision, recall, or accuracy values not yet established
- measured business ROI not yet documented
- reduced rework percentages without a published method and data
- broad external adoption
- model-agnostic reliability
- automated correction or guaranteed prompt improvement

### Do Not Imply
- that the tool rewrites prompts into a correct final form
- that all ambiguity can be solved by public web data
- that every `/照` output is actionable
- that ambiguity detection equals truth

### Do Not Show
- unsanitized merchant or internal names
- internal URLs or metadata
- screenshots with hidden identifiers
- exact internal prompt excerpts if they can be traced back

### Tone to Avoid
- hype-heavy AI guru language
- benchmark theater
- faux-scientific certainty
- product-marketing abstraction detached from cases

## 15. Acceptance Criteria

The promotional package is acceptable only if all conditions below are met.

### A. Strategic Fit
- The package clearly optimizes for trying `/照`, not just understanding it
- The README and one long-form artifact work as a pair
- The cases + method dual track is visible

### B. Trustworthiness
- No fabricated numbers or unsupported claims
- No precision or accuracy claims before a documented baseline exists
- At least one explicit statement of method limitation appears
- Reflection framing is preserved throughout

### C. Sanitization
- All merchant and internal names are removed or generalized
- All internal links and metadata are stripped
- All case excerpts pass a manual anonymity check

### D. Message Clarity
- The main promise is understandable in under 15 seconds on the README
- The try/install path is visible without deep scrolling
- The difference from prompt rewriting or scoring tools is explicit
- The copy is concise and information-dense: no section exists only for atmosphere

### E. Case Quality
- 6 real-source-derived cases are selected for the long-form artifact
- Each case is understandable without internal context
- Each case includes ambiguity-before and reflection-after
- The README uses 1 flagship case drawn from the same case pool

### F. CTA Quality
- The README includes a primary CTA and a secondary star/follow CTA
- The long-form artifact includes at least two CTA moments
- CTA language emphasizes trial over commitment

### G. Tone Discipline
- No hype claims
- No vague 'better prompts' framing without specifics
- No managerial abstraction without concrete cases
