## 1. 产品定义

### 1.1 一句话定位

**当产出物所有者认为内容已达到自己认知内的最优水平、准备交给下一个人或 agent 时，准确识别并展示那些仍未被看见、但会在下游引发高成本返工的关键 gap。**

### 1.2 它是什么

FindGap 是一个 **交接点 Gap 校对系统**。它以 skill（技能文件）为主要载体，运行在 Claude Code、Codex CLI 等 AI agent 环境中。它不介入每一轮人机对话，也不要求用户在每次与 agent 交互前都做检查；它服务的是一个更值钱的时刻：**当所有者已经认为当前产出物没有明显问题，准备把它交给下一个人或下一个 agent 继续推进时**。这时，FindGap 以自检或他检的方式，优先准确识别那些 agent 在执行前难以主动发现、但会在下游造成高返工成本的结构性 gap，并将这些 gap 以可追溯、可验证、可分级的方式展示出来。联网查行业数据和最佳实践只作为辅助依据，用于帮助用户理解为什么这是一个真实的 gap，以及通常应如何补齐；产品主价值始终是 **交接前的 gap 识别与校对**，而不是替用户生成答案。

### 1.3 它不是什么

- 不是 prompt 优化器（不帮你重写 prompt）
- 不是 AI 质量平台（不覆盖执行与交付阶段）
- 不是通用写作建议工具（只扫发给 AI 的任务/方案/意图）
- 不替用户做最终决策（给方向、给依据，用户自己判断）

### 1.4 核心用户

重度 AI 协作者：经常给 AI 下任务的开发者、架构师、产品经理、技术负责人。不限行业、不限 agent 平台。

### 1.5 核心承诺

**在交给下一个人或 agent 之前，准确识别并校对那些会在下游引发返工的高成本 gap。**

### 1.6 结果导向价值主张

FindGap 识别 gap 的目的，不是为了“指出问题”本身，而是为了：

1. **减少返工**：在认知开始流转前，把错误、不完整或默认前提拦下来。
2. **最大化 ROI**：优先识别那些返工代价最高、下游 blocker 最强、人工最难低成本检查的 gap。
3. **促进自我成长**：帮助所有者和接头人在一次次交接校对中，逐步形成更结果导向的思维方式，而不是只停留在“我觉得已经差不多了”。
4. **提升价值输出**：让产出物不仅“看起来完整”，而是真正更容易被下游正确理解、正确执行、正确验收。

FindGap 的最终目标不是增加一个检查动作，而是帮助用户在关键交接点上，用更高 ROI 的方式提升价值输出质量。

### 1.7 产品原则

1. **识别优先**：先把 gap 准确识别出来，再谈如何补齐。
2. **展示优先**：核心交付是把 gap 说清、标清、分级、可追溯地展示出来。
3. **高成本优先**：优先识别 agent 难以预判、人工识别成本高、下游返工代价大的 gap。
4. **交接点聚焦**：只在产出物准备进入下一个流水线节点时做校对，不在每一轮人机交互中打断用户。
5. **垂直场景深挖**：优先在高复杂度、强约束、强协作的专业场景里把识别准确率做深，而不是追求泛化广度。
6. **最佳实践为辅**：行业做法和实践只用于辅助理解 gap，不喧宾夺主。
7. **方法必须准确**：规则边界、证据层级、输出原文引用必须可校验，避免把产品做成泛泛建议器。

### 1.8 核心场景

FindGap 1.0.0 只聚焦一个细分但高价值的场景：

**当产出物所有者已经认为内容在自己当前认知内没有明显问题，准备把它交给下一个流水线接头人继续推进时，做一次交接前的认知校对。**

这里的“下一个接头人”既可以是人，也可以是 agent。FindGap 的任务不是在多轮协作过程中频繁打断用户，而是在**认知即将流转**的那个节点，把那些仍未被意识到、但会在下游放大成返工的 gap 揪出来。

### 1.9 双模式

#### A. Self-check（所有者自检）
当所有者认为产出物已经可交接时，FindGap 帮他识别：
- 哪些默认前提只存在于自己脑中，没有显式写出
- 哪些地方自己以为说清了，但下游未必能按同样方式理解
- 哪些 gap 已经超出自己当前认知范围，因此不会再主动要求 agent 帮忙优化

#### B. Receiver-check（接头人他检）
当下游接头人准备接手产出物继续推进时，FindGap 帮他识别：
- 如果现在开工，会卡在哪些信息缺口上
- 哪些地方会迫使自己默认补脑，从而带着错误认知往下做
- 哪些 gap 会在执行、联调、验收或跨角色协作中放大成返工

### 1.10 触发方式

`/findgap` `/fg` `/照` `照一下`

## 2. 1.0.0 功能边界

### 2.1 1.0.0 要解决的核心问题

FindGap 1.0.0 不追求回答“AI 应该怎么做”，也不追求在多轮协作过程里频繁介入，而聚焦回答三个更前置的问题：

1. **当所有者已经觉得这份产物可以交接时，里面是否仍然有高成本 gap？**
2. **这些 gap 为什么会在下游造成误解、卡点或返工？**
3. **这些 gap 应该如何被准确地展示给所有者或接头人，以便及时澄清和矫正？**

因此，1.0.0 的首要目标不是生成更多建议，而是建立一套**面向交接点的、科学、有效、可验证的 gap 识别与校对系统**。

### 2.2 对外主打的 4 类高成本专业 gap

1.0.0 对外不讲 11 条规则，讲 4 类高成本专业 gap。它们有共同特征：**agent 在执行前难以安全补全，所有者在当前认知内也往往不会主动补问，而一旦带着这些缺口交接，下游就会被迫在错误或不完整认知上继续工作。**

| 类别 | 解决什么 | 对应内部规则 |
|------|---------|-------------|
| **完成/验收缺口** | 做到什么算完成、验收标准缺失、可观察输出不明确 | R-DOD |
| **边界/失败路径缺口** | 异常、回滚、降级、受影响方、排除项没写 | V-STAKE, G-NOGO |
| **术语/定义缺口** | 同名异义、定义不一致、缺权威来源 | V-NAME, I-SSOT |
| **方法/约束/依赖缺口** | 性能、合规、系统边界、关键依赖和决策理由未锁定 | S-PERF, S-QUANT, S-NFR, I-ADR |

其余规则（G-WHY, V-LAYER）继续在内部驱动检测，但不作为对外交接场景的第一梯队。

### 2.3 系统模块架构

1.0.0 围绕“**检测 + 展示**”建设最小闭环。FindGap 从 skill 升级为模块化系统，但所有模块都服务于同一件事：**把高成本专业 gap 找准、标清、分级并可追溯地展示出来。**

```
┌─────────────────────────────────────────────────┐
│                FindGap 1.0.0 系统                  │
├─────────────────────────────────────────────────┤
│                                                 │
│  ┌───────────┐    ┌───────────┐    ┌──────────┐ │
│  │ 输入解析   │───▶│ Gap 检测  │───▶│ 严重度   │ │
│  │ 模块      │    │ 模块      │    │ 判定模块  │ │
│  └───────────┘    └───────────┘    └──────────┘ │
│        │                │               │       │
│        ▼                ▼               ▼       │
│  ┌───────────┐    ┌───────────┐    ┌──────────┐ │
│  │ 受众检测   │    │ 依据处理   │    │ 不确定性  │ │
│  │ 子模块    │    │ 模块      │    │ 处理模块  │ │
│  └───────────┘    └───────────┘    └──────────┘ │
│                         │               │       │
│                         ▼               ▼       │
│                   ┌───────────┐    ┌──────────┐ │
│                   │ 补齐方向   │    │ 结果展示  │ │
│                   │ 指引模块   │    │ 模块     │ │
│                   └───────────┘    └──────────┘ │
│                                                 │
│  ┌─────────────────────────────────────────────┐ │

└─────────────────────────────────────────────────┘
```

#### 模块职责

| 模块 | 输入 | 输出 | 职责 |
|------|------|------|------|
| **输入解析** | 用户原始文本 | 结构化输入对象 | 识别文本类型（一句话需求/技术方案/PRD 片段/决策请求） |
| **受众检测** | 结构化输入 | 受众标签 + 输出策略 | 判断受众是决策者/评审者/作者本人，决定展示详略 |
| **Gap 检测** | 结构化输入 | gap 候选列表 | 11 条规则逐条扫描，产出原始命中 |
| **严重度判定** | gap 候选列表 | 带严重度的 gap 列表 | 致命/阻塞/可延后 三级判定 |
| **依据处理** | gap 列表 | 带依据类型的 gap 列表 | 区分公开证据、上下文推理、当前不可判定三类判断依据 |
| **不确定性处理** | 缺证据或边界模糊的 gap | 带边界标记的 gap | 对信息不足、边界不清、无法安全判断的情况做显式标注 |
| **补齐方向指引** | 带依据的 gap 列表 | 带辅助指引的 gap 列表 | 提供可选补齐方向，作为识别结果的辅助信息，不替用户下结论 |
| **结果展示** | 完整 gap 列表 | Flat List 输出 | 按严重度排序，固定展示原文锚点、缺口、风险、依据类型和补齐方向 |
| **评测与样本** | 历史运行记录 | baseline 报告 | 精度、采纳率、改写率、展示质量统计 |

### 2.4 三节点执行流程（不变）

```
① 找交接缺口（按 self-check / receiver-check 视角扫描 11 规则并判定严重度）
  ↓
② 校验依据（公开证据 / 上下文推理 / 信息不足）
  ↓
③ 展示并校对（severity + 原文锚点 + 缺失信息 + 下游风险 + 依据类型 + 可选补齐方向）
```

### 2.5 1.0.0 新增能力（相对 v0.9）

| 新增项 | 说明 |
|--------|------|
| **规则冻结** | 连续 2 版本精度无回归（各规则 precision 波动 ≤5%） |
| **双模式支持** | 支持 Self-check（所有者自检）和 Receiver-check（接头人他检）两种交接视角 |
| **交接闭环指标** | 新增：是否改写产出物、是否采纳命中项、是否减少下游 blocker、owner/receiver 发现差异 |
| **CLI runner** | `findgap check <file>` / `findgap eval --suite <dir>` / `findgap report` |
| **eval runner** | 批量执行样本、生成结构化报告、记录命中/改写/采纳结果 |
| **软件说明书** | 完整的用户手册、系统架构说明、模块接口定义 |
| **垂直领域样本分层** | 按场景分层验证交接点 gap 识别效果，优先覆盖高复杂度、强约束、强协作的专业文本 |
| **展示质量校验** | 强化原文引用、严重度分级、依据类型、输出一致性等展示质量检查 |
| **高成本专业 gap 样本库** | 建立 agent 难以预判、人工识别成本高、返工代价大的专业缺口样本库 |

### 2.6 1.0.0 不做（明确排除）

- 不扫 PRD/PR/对话记录/AI 已交付的输出（属 v2.0+ 方向决策）
- 不做实时 CI/CD 集成（属 v2.0+）
- 不做多语言 UI（1.0.0 以中英双语 CLI 为主）
- 不做 SaaS 化（1.0.0 是本地 skill + CLI）
- 不替用户改写文档
- 不给 PASS/FAIL 二值判定（v0.6 已砍）
- 不做跨模型比较、模型影响归因或模型选择建议
- 不做总体 prompt 评分器或泛化 request quality 平台
- 不做事后输出审查（属 FindAction / FindMiss 的未来能力面）

### 2.7 红线（5 条）

1. 三节点上限——不多于找缺口 / 校验依据 / 展示缺口三步
2. 不落盘——运行时不维护任何静态数据库，行业数据每次现查
3. URL 真实——编造 URL 一次 = 信用归零
4. 补齐方向从属于缺口识别——提供决策依据和可选做法，但不替用户做最终选择
5. 方法可追溯——每条 gap 必须明确其判断依据属于公开证据、上下文推理或信息不足，禁止用泛化 best practice 伪装确定性

### 3.3 软著核心优势（原创性亮点）

以下内容构成 FindGap 的核心原创贡献，对软著申请最有利：

1. **11 条 gap 检测规则体系**
   - 从 171 条根因中经三维交集（频率×代价×可拦截）筛选出 11 条
   - 每条规则有明确的触发条件、边界定义、相邻规则抑制逻辑
   - 经 20 轮真实验证，整体 precision 0.9805

2. **三层证据兜底机制**
   - Tier 1：WebSearch ≥2 条可点击来源互证
   - Tier 2：搜不到时基于文档现状与领域常识推理
   - Tier 3：信息不足时明确标注
   - 设计原则：永远不停在"没数据"

3. **受众感知输出策略**
   - 根据输入信号自动判断受众类型（决策者/评审者/作者）
   - 不同受众对应不同输出详略策略

4. **严重度判定与优化建议生成**
   - 致命/阻塞/可延后 三级判定
   - 每条致命和阻塞 gap 必须附优化方案与决策依据

5. **数据驱动的 dogfood 验证体系**
   - 判定协议、baseline 报告、分层统计、matched-strata 比较
   - 从 v0.1 到 v0.9 共 9 轮迭代，累计砍掉 160+ 条规则和多个过度设计

## 6. 竞争力评估

### 6.1 FindGap 解决的是真实问题吗？

**是。** 但它解决的不是泛泛的“prompt 写得好不好”，而是**产物流转前那一刻最容易被忽视、但最容易把错误认知带进下游的交接缺口**。证据如下：

- 20 轮真实验证，precision 0.9805
- 最高频的 4 类高成本专业 gap（完成/验收、边界/失败路径、术语/定义、方法/约束/依赖）在样本中反复出现
- 这些 gap 往往发生在所有者已经认为“差不多了”的时刻，不会在继续追问 agent 的过程中自然暴露
- 一旦带着这些 gap 进入交接，下游会被迫在错误或不完整认知上继续工作，返工代价高

### 6.2 它和现有产品的区别是什么？

| 对比 | FindGap | Prompt Optimizer 类 | Code Review 类 |
|------|---------|---------------------|---------------|
| 核心时机 | 交接前 / 流转前 | 写 prompt 时 | 交付后 |
| 核心目标 | 识别并校对高成本交接 gap | 改写 prompt | 找代码问题 |
| 主战场 | owner 已自认完整、准备 handoff 的产物 | 表达更顺、更完整 | 已写出的代码或结果 |
| 检查视角 | self-check + receiver-check | 单一作者视角 | reviewer 视角 |
| 是否替你决策 | 不替 | 往往替你重写 | 告诉你改 |
| 是否要求原文锚点与依据类型 | 是 | 通常否 | 部分 |
| 是否处理“当前不可判定” | 是 | 通常否 | 通常否 |
| 是否有 baseline | 是 | 否 | 部分 |

### 6.3 它的护城河是什么？

1. **交接点高成本 gap 的识别精度**：171 → 11 的筛选过程和验证数据是核心 IP
2. **双视角校对能力**：同一产物可从 owner self-check 和 receiver-check 两个角度发现不同 gap
3. **可追溯的展示契约**：原文锚点、严重度、依据类型、缺失信息、下游风险、补齐方向的固定结构降低误读
4. **诚实处理不确定性**：能区分公开证据、上下文推理和信息不足，不用泛化 best practice 伪装确定性
5. **dogfood 验证体系**：精度、采纳、改写、双视角差异与展示质量的闭环数据

## 7. 设计决策记录

### ADR-1：为什么 1.0.0 只推 FindGap，不推三件套

- **决策**：TrustChain 收口到 FindGap 作为唯一 1.0.0 产品
- **理由**：用户安装一个 skill 的意愿远高于三个；FindGap 痛点最高频、ROI 最直观；三产品并列分散叙事和开发资源
- **替代方案**：三件套同时推（否决，因为 FindAction/FindMiss 场景不够高频、心智成本高、容易互相重叠）

### ADR-2：为什么补 CLI 而不是做 Web UI

- **决策**：1.0.0 补 CLI runner
- **理由**：CLI 最贴近目标用户（AI coding 用户）的工作环境；CLI 同时解决软件化表达、演示、测试、软著材料完整性；Web UI 投入产出比不合适
- **替代方案**：Web UI（否决，因为目标用户在终端工作，Web UI 增加部署和维护成本）

### ADR-3：为什么不删 FindAction/FindMiss 仓库

- **决策**：降级而非删除
- **理由**：delete 不可逆，品牌痕迹丢失；真正的问题是叙事失焦，不是仓库存在；archive 可以达到收口效果
- **替代方案**：delete（否决，损失大于收益）；private（否决，对开源社区信号不好）
