## 背景

该 PRD 同时处理两类银行卡支付治理问题：一类是同一订单短时间内重复扣款后的自动识别与自动原路退款/冲正，另一类是渠道成功但 PP 失败时，允许运营通过“修单”按钮查询渠道状态并把失败订单修成成功，目标是减少重复支付损失与人工状态修复成本。

## 节点 ② 查

- I-SSOT: PayPal、Authorize.net 等公开资料都明确区分 void / authorization reversal / refund，并且与 auth-only、capture、sale 的可用时点绑定。当前 PRD 把“退款 / 冲正 / 撤销 / 原路退款”混写，同时范围覆盖 auth/capture、sales、one-time payment，却没映射每类交易对应哪一种资金动作，会直接把状态机、接口契约和资金语义拆成多个版本。
- V-STAKE: Adyen Refund、Primer Webhooks、Peach Payments 等资料都强调退款、查询补偿和最终到账不是一步完成，会出现 pending、处理中、失败重试、异步 webhook 与对账回补。当前 PRD 只写 happy path，没有覆盖查询超时、退款失败、部分退款、人工已处理、重复点击修单和回调竞态等失败闭环。
- R-DOD / S-QUANT: PayPal、Adyen、Antom 的 idempotency 文档都把重复支付防控建立在 request id、idempotency key、安全重试和明确窗口上。当前“短时间内”“重复支付”“状态不一致”都没有秒/分钟边界、匹配键或排除项，也没有“修单完成”的可验证验收口径，不同团队很可能各自实现不同规则。
- V-NAME / I-ADR / G-NOGO / G-WHY: 公开支付资料通常把 merchant order、gateway transaction、acquirer result、capture/refund status 分层定义，也通常把 query 视作补偿手段之一，而不是无条件终态来源。当前“修单”“PP失败”“渠道成功”“成功”都未自解释；同时文档没声明自动退款与人工修单的适用边界、优先级、不做范围，也没明确主目标究竟是降客诉、减人工、缩短退款时延还是减少对账差异，因此范围、术语和决策依据都存在真实协作缺口。

## 节点 ③ 亮摘要

致命 2 条：文档把“退款 / 冲正 / 撤销 / 原路退款”混写，又同时覆盖 auth/capture、sales、one-time payment，却没定义各自对应的资金动作，状态机与接口契约无法共享单一事实源（I-SSOT）；自动退款与修单都只写 happy path，没有退款失败、处理中、查询超时、异步回补、人工已处理和并发竞态的补偿闭环，后续极易返工（V-STAKE）。

阻塞 4 条：没有定义重复支付识别规则、时间窗、匹配键和“修单完成”的验收口径，不同团队会各自实现不同去重与修复逻辑（R-DOD）；“修单 / PP失败 / 渠道成功 / 成功”都不是文中已建好的统一术语，运营、支付后端和风控很可能读出不同状态层级（V-NAME）；自动退款与人工修单的适用场景、优先级和不做范围没有声明，职责边界容易互相覆盖（G-NOGO）；文档直接规定“查渠道后把 PP 失败改成功”，却没解释为什么 query 可以作为单向终态依据，也没说明为何不支持处理中、关闭或待退款等其他终态（I-ADR）。

可延后 2 条：背景说明了现象，但没有进一步明确这项需求主要追求的是降客诉、减工单、缩短退款时延还是减少对账差异，因此收益口径不够聚焦（G-WHY）；“短时间内”“重复支付”“状态不一致”没有量化边界，证据强度略弱于前述结构性问题，但仍会影响多系统一致实现（S-QUANT）。
