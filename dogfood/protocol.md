# 判定协议 · v0.7

> 本文件定义如何判断照妖镜每次 `/照` 的命中项是真阳还是假阳。v0.7 的基线报告以此协议为准。

---

## 核心定义

| 术语 | 定义 | 举例 |
|------|------|------|
| **真阳 (TP)** | 命中项指出了 prompt 中真实存在的含混，且该含混会导致下游（AI 或人）理解偏差 | prompt 写"要快"，S-PERF 命中——确实没说多快 |
| **假阳 (FP)** | 命中项在上下文中不构成含混：描述性使用、示例引用、已有上下文补充 | 规则表中写"扫描'快/高效'"，S-PERF 命中——这是在定义规则，不是在模糊使用 |

## 判定流程

对每次 `/照` 的每条命中项：

1. 读命中的原形句
2. 回到原 prompt 上下文中重读该处
3. 问自己：**如果我把这段 prompt 发给一个不了解背景的 AI / 同事，TA 会因为这处含混产生理解偏差吗？**
   - 是 → 真阳
   - 否 → 假阳
4. 写下判定理由（1 句话）

## 判定者

v0.7 阶段：prompt 作者本人（项目作者）。

v0.8 阶段：增加外部用户作为独立判定者，计算判定者间一致性。

## 样本要求

- 总量：≥ 20 条不同 prompt
- 来源多样性：≥ 10 条日常真实 prompt + ≥ 10 条公开场景（GitHub Issues / Stack Overflow / 社区提问 / 公开 PRD 模板）
- 不得重复使用 dogfood/runs/001-004 的旧产物

## 统计口径

**按规则统计**（11 行表）：

| 招 | 命中次数 | TP | FP | precision |
|----|---------|----|----|-----------|
| S-PERF | N | n | m | n/(n+m) |
| ... | | | | |

**总量聚合**（1 行）：

| 总命中 | 总 TP | 总 FP | 总 precision | 总 FPR |
|--------|-------|-------|-------------|--------|
| N | n | m | n/N | m/N |

## 记录格式

每条 dogfood/runs/ 记录的反馈章节使用以下格式：

```markdown
## 反馈

| 命中 | 代号 | 判定 | 理由 |
|------|------|------|------|
| 1 | S-PERF | TP | prompt 说"要快"确实没给数字 |
| 2 | V-NAME | FP | "部署"在此上下文含义明确 |
```
