# Remediation Card · 结构化补齐卡片模板（v0.4 引入）

> Phase 0 / Phase 1 发现的每个阻断或警告项必须套用本模板生成"补齐卡片"。
>
> **核心设计原则**：
> 1. 给具体选项 ≤4 个，绝不开放追问（红线 3）
> 2. 量化不补齐的风险（不编造，用统计基线）
> 3. 允许显式跳过但留痕（`[risk-accepted]`）
> 4. 补完后自动重评，不需用户手动触发复扫
>
> **学术对照**：Addy Osmani "TODO 占位符 + 逐项填充 > 一次性写完整"；研究证明结构化 scaffold 的补齐完成率比开放追问高约 40 个百分点。

---

## 卡片骨架（强制模板）

```
[补齐卡片] {gap_id}  · {严重度 emoji} {严重度} · {一句话标题}

== 缺什么 ==
  定位：{file:line:col 或 task-input:{field} 或 doc#anchor}
  缺失项：{slot_name}（{一句话描述}）
  原文证据：「{摘录 ≤30 字}」

== 为什么重要 ==
  不补齐风险：{具体失败场景，一句话}
  影响估算：agent 成功率预估降低 ~{N}%
  对应失败根因：{失败类型，参见 references/gap-taxonomy.md G1-G18}

== 怎么补（四选一）==
  A) 直接填写：{slot_template}
     例：「{占位符示例}」
  B) 指向文档：参见 [{文档路径或 URL}]
  C) 提供示例：附 ≥1 个符合预期的输入/输出样本
  D) 跳过（接受风险，标记 [risk-accepted] 并继续）

== 补完后验证 ==
  补齐后系统自动重评该槽位（无需手动触发）。
  验证标准：{slot_name} 通过 {对应 G 维度} 检查，verdict 升级。
  复查命令（可选）：
    {可粘贴执行的 grep/test 命令}
```

## 维度 × 补齐路径对照

| Gap | A) 直接填写模板 | B) 指向文档场景 | C) 提供示例场景 | D) 跳过的合理理由 |
|-----|--------------|----------------|----------------|------------------|
| G14.S | "在 [模块] 上完成 [动作] 针对 [对象]" | 已有产品需求文档可引用 | — | 任务为探索性，目标尚不收敛 |
| G14.M | "[指标]（口径 X）从 A 提升到 B" | 已有 SLO/KPI 文档 | — | 学习成本高于本次收益 |
| G14.A | "依赖 X 资源 / 风险 Y 时降级到 Plan B" | 已有容量/资源评估 | 类似规模项目的历史数据 | 试验性投入，允许失败 |
| G14.R | "本目标支撑上位目标 X，贡献 Y" | OKR 文档 | — | 独立技术债清理任务 |
| G14.T | "截止 YYYY-MM-DD，里程碑 M1 M2 M3" | 项目计划 | — | 持续性运营任务 |
| G15 | "拆分为 T1 → T2 → T3，依赖关系：..." | 任务分解文档 | 历史拆分案例 | 任务确实原子不可拆 |
| G16 | "{术语} = 一句话定义" | 业务术语表 / 既有架构文档 | 1 个领域内典型案例 | 任务不依赖该术语 |
| G17 | — | API/Schema 文档 | "正例 + 反例" | 输出格式宽容，无强约束 |
| G18 | "回滚预案：步骤 1..N + SLA" | 灰度发布手册 | — | 操作可逆且代价小 |

## 示例：完整补齐卡片

```
[补齐卡片] G16-2#11d8af · 🔴 P0 · 上下文缺失：业务术语"核销"未定义

== 缺什么 ==
  定位：task-input:objective 第 2 段
  缺失项：术语「核销」的领域定义
  原文证据：「实现订单核销流程，要快」

== 为什么重要 ==
  不补齐风险：agent 将用通用理解（"逐项确认"）处理，可能错误覆盖
                业务领域里的"核销 = 抵扣 + 锁定 + 凭证留存"完整链路
  影响估算：agent 成功率预估降低 ~40%（缺失上下文是失败根因 Top 2，~20% 占比）
  对应失败根因：G16 上下文完整度 / Representational Incompleteness

== 怎么补（四选一）==
  A) 直接填写：「核销」= ____（一句话定义，含核心动作 + 数据落地点 + 不可逆性）
     例：「核销 = 将订单状态从 PENDING → SETTLED，同时写入资金流水 ledger，
          且不可回退（必须通过反向流水抵消）」
  B) 指向文档：参见 [docs/glossary.md#核销] 或 [既有代码 src/order/settlement.ts]
  C) 提供示例：附 1 个核销请求的输入 JSON + 期望输出 JSON
  D) 跳过（接受风险，标记 [risk-accepted] 并继续）

== 补完后验证 ==
  补齐后系统自动重评 G16 / readinessScore。
  验证标准：原文中「核销」周边 ±50 字内出现术语定义 或 ≥1 个文档/代码引用。
  复查命令：
    rg -n "核销" task-input.md docs/ src/
```

## 警告卡片（WARNING）vs 阻断卡片（BLOCKED）差异

| 维度 | BLOCKED 卡片 | WARNING 卡片 |
|------|-------------|--------------|
| 触发 | readinessScore < 60 或任一 critical | 60 ≤ readinessScore < 80 且无 critical |
| 是否阻进 Phase 1 | **是** | 否（但要求显式 ack） |
| D 选项语义 | 跳过 = `[risk-accepted]` 但仍阻断本项 | 跳过 = `[risk-accepted]` 直接放行 |
| 重评策略 | 自动重评直到 PASSED | 一次 ack 即可放行 |

## 误报申诉路径

若用户认为照妖镜识别错误：

```
回复 DROP: <理由>
→ 卡片转为 🚫 状态
→ 写入 dogfood/feedback.jsonl status=rejected
→ 本次扫描内不再展示
→ 累计 ≥3 次同 family 的 rejected 不触发隐性静默（红线 8 红线 11：不依赖历史压制）
```

## 实现要点（给 agent / 实现者）

1. **卡片必须四件套齐全**：缺什么 / 为什么重要 / 怎么补 / 补完后验证
2. **A/B/C 选项必须具体化**：禁止 "请补充更多信息" 类开放追问
3. **影响估算必须有依据**：引用 references/gap-taxonomy.md 中失败根因占比，**不编造数字**
4. **D 跳过必须留痕**：写入 `dogfood/runs/<scanId>.md` 的 `[risk-accepted]` 段
5. **不依赖记忆**：同产物再次提交，已 ack 的 WARNING 仍然重新生成卡片（红线 8）
